# ADR 0002: Canonical Workout Data Model

**Status:** Proposed  
**Date:** 2026-07-05  
**Issue:** #120  
**Depends on for persistence:** #119  
**Follow-up implementation:** #123

## Context

GymTrack currently represents one workout in several overlapping forms:

- `NoteEntity`, where the note timestamp is both the primary key and workout identity;
- encoded note text containing exercise structure, flags, and timing metadata;
- `ExerciseEntity` and `SetEntity` rows generated by reparsing note text;
- statistics that may query set rows or parse note text again;
- CSV export generated from the note representation.

The current `SetEntity.workoutId` is the note timestamp. Sets link directly to exercises, so there is no persisted workout-exercise occurrence, no explicit exercise order, and no explicit set order. Modifier, equipment brand, and timing values are duplicated on each parsed set even though they originate from an exercise header.

These representations can disagree. They also make edits, migrations, import/export, statistics, Creator OS summaries, and future synchronization difficult to reason about.

## Decision

Normalized typed data becomes the canonical persisted workout representation.

The note-like editor remains a product interface, but after validation its state is saved as one typed workout aggregate:

```text
Workout
  1 -> many WorkoutExercise
  1 -> many WorkoutSet through WorkoutExercise

Exercise
  1 -> many WorkoutExercise
  1 -> many ExerciseAlias

Category
  1 -> many Workout
```

Raw legacy note text is retained temporarily for migration verification, recovery, and rollback. It is not the canonical source for history, statistics, export, or Creator OS summaries after a workout has migrated successfully.

## Canonical domain model

The executable pure Kotlin model is defined in:

```text
app/src/main/java/com/example/gymtrack/domain/model/CanonicalWorkout.kt
```

### Workout

| Field | Purpose |
|---|---|
| `id` | Stable generated identifier independent from time. |
| `startedAtEpochMillis` | Actual workout start/display time. |
| `endedAtEpochMillis` | Optional completion time. |
| `categoryId` | Optional stable category reference. |
| `title` | Optional display title. |
| `learnings` | Workout-level learning/note. |
| `status` | `DRAFT`, `PARTIAL`, or `COMPLETED`. |
| `createdAtEpochMillis` | Record creation timestamp. |
| `updatedAtEpochMillis` | Last persisted modification timestamp. |
| `legacyCompatibility` | Temporary migration/recovery payload only. |

### WorkoutExercise

A workout exercise represents one occurrence of an exercise inside one workout. Repeating the same exercise later in a workout creates another occurrence.

| Field | Purpose |
|---|---|
| `id` | Stable occurrence identifier. |
| `workoutId` | Parent workout. |
| `exerciseId` | Canonical exercise. |
| `position` | Explicit workout ordering. |
| `mode` | `BILATERAL`, `UNILATERAL`, or `SUPERSET`. |
| `modifier` | Grip, stance, equipment type, or other current parser modifier. |
| `equipmentBrand` | Current parser brand output. |
| structured timing | Offset or absolute timestamp when confidently parsed. |
| legacy timing text | Temporary exact text retained when conversion is ambiguous. |

Modifier and equipment brand belong to the workout-exercise occurrence, not every set, because the current parser derives them from the exercise header and repeats them onto all sets.

### WorkoutSet

| Field | Purpose |
|---|---|
| `id` | Stable set identifier. |
| `workoutExerciseId` | Parent workout-exercise occurrence. |
| `position` | Explicit order within the exercise occurrence. |
| `repetitions` | Optional repetition count. |
| `weight` and `weightUnit` | Optional load with explicit unit. |
| `durationSeconds` | Optional duration metric. |
| `distanceMeters` | Optional distance metric. |
| `performedAtOffsetSeconds` | Optional time relative to workout start. |
| `rpe` / `rir` | Optional future effort fields. |

A set must contain repetitions, duration, or distance. Weight cannot exist without an explicit unit. This prevents the current ambiguity where cardio duration may be stored in a repetition field and kilograms/pounds may be implicit.

### Exercise

Exercises use stable identifiers and a canonical name. Aliases are modeled as individual values rather than a comma-delimited string. An optional parent reference supports variations without changing workout history.

### Category

Categories use stable identifiers. Rename and color changes do not change workout references. Built-in categories receive deterministic IDs. Custom categories receive generated IDs.

## Proposed Room schema

The Room entities will be introduced by #123 after the explicit migration-safety work in #119 is merged.

### `workouts`

```text
id TEXT PRIMARY KEY
legacy_timestamp INTEGER UNIQUE NULL
started_at INTEGER NOT NULL
ended_at INTEGER NULL
category_id TEXT NULL REFERENCES categories(id) ON DELETE SET NULL
title TEXT NOT NULL
learnings TEXT NOT NULL
status TEXT NOT NULL
raw_draft_text TEXT NULL
legacy_migration_status TEXT NULL
legacy_migration_message TEXT NULL
created_at INTEGER NOT NULL
updated_at INTEGER NOT NULL
```

Indexes:

- `started_at`
- `category_id`
- `status`
- unique nullable `legacy_timestamp`

### `categories`

```text
id TEXT PRIMARY KEY
name TEXT NOT NULL
color_argb INTEGER NOT NULL
position INTEGER NOT NULL
is_built_in INTEGER NOT NULL
is_archived INTEGER NOT NULL
```

The database should not require category names to be globally unique. Historical categories can remain archived while a user creates another category with a similar display name.

### `exercises`

```text
id TEXT PRIMARY KEY
canonical_name TEXT NOT NULL
parent_exercise_id TEXT NULL REFERENCES exercises(id) ON DELETE SET NULL
muscle_group TEXT NULL
created_at INTEGER NOT NULL
updated_at INTEGER NOT NULL
```

Use a normalized-name index for lookup, but do not use display name as identity.

### `exercise_aliases`

```text
id TEXT PRIMARY KEY
exercise_id TEXT NOT NULL REFERENCES exercises(id) ON DELETE CASCADE
normalized_alias TEXT NOT NULL
original_alias TEXT NOT NULL
UNIQUE(exercise_id, normalized_alias)
```

### `workout_exercises`

```text
id TEXT PRIMARY KEY
workout_id TEXT NOT NULL REFERENCES workouts(id) ON DELETE CASCADE
exercise_id TEXT NOT NULL REFERENCES exercises(id) ON DELETE RESTRICT
position INTEGER NOT NULL
mode TEXT NOT NULL
modifier TEXT NULL
equipment_brand TEXT NULL
started_at_offset_seconds INTEGER NULL
started_at INTEGER NULL
legacy_relative_time_text TEXT NULL
legacy_absolute_time_text TEXT NULL
UNIQUE(workout_id, position)
```

Indexes:

- `workout_id`
- `exercise_id`
- `(workout_id, exercise_id)`

### `workout_sets`

```text
id TEXT PRIMARY KEY
workout_exercise_id TEXT NOT NULL REFERENCES workout_exercises(id) ON DELETE CASCADE
position INTEGER NOT NULL
repetitions INTEGER NULL
weight REAL NULL
weight_unit TEXT NULL
duration_seconds INTEGER NULL
distance_meters REAL NULL
performed_at_offset_seconds INTEGER NULL
rpe REAL NULL
rir REAL NULL
UNIQUE(workout_exercise_id, position)
```

Index:

- `workout_exercise_id`

## Aggregate invariants

Before a transaction is persisted:

- every ID is non-blank and stable;
- workout identity is independent from workout timestamps;
- end time cannot precede start time;
- every workout exercise belongs to the workout;
- workout-exercise positions are unique within the workout;
- every set references a workout exercise in the aggregate;
- set positions are unique within each workout exercise;
- numeric values cannot be negative;
- optional metrics remain null instead of being inferred.

These rules are covered by pure JVM tests and should also be enforced through Room constraints where possible.

## Migration strategy

### Prerequisite

Do not add the new tables to production until #119 removes destructive migration fallback and provides repeatable Room migration tests.

### Stage 1: Add schema beside legacy tables

Add the canonical tables without removing `notes`, `exercises`, or `sets`. Existing application reads remain unchanged.

### Stage 2: Backfill workouts

For each `NoteEntity`:

1. Generate a stable workout ID.
2. Copy `timestamp` into `legacy_timestamp` and `started_at`.
3. Copy title and learnings.
4. Resolve or create a stable category ID from category name/color.
5. Retain exact note text in `raw_draft_text`.
6. Mark migration status `PENDING` until the aggregate verifies successfully.

Generated IDs are persisted once; they must not be regenerated on every launch.

### Stage 3: Backfill workout structure

Parse legacy note text once at the migration boundary.

- Preserve exercise order from note order.
- Create a new `WorkoutExercise` for every explicit occurrence, even when the same canonical exercise appears twice.
- Preserve bilateral, unilateral, and superset mode.
- Move modifier and equipment brand to the workout-exercise occurrence.
- Preserve set order from note order.
- Parse timing into structured values only when unambiguous.
- Retain original timing text during the compatibility period.

Current `SetEntity` rows are verification data, not the sole migration input, because they do not preserve occurrence ordering and may have been rebuilt by a different parser version.

### Ambiguous or failed migration

Do not silently invent data.

When parsing fails or legacy rows disagree:

- keep the original note untouched;
- mark the workout `NEEDS_REVIEW`;
- record a compact migration message;
- exclude uncertain derived values from canonical statistics until resolved;
- continue allowing legacy display/recovery for that workout.

### Verification

For representative fixtures compare:

- workout counts;
- exercise occurrence counts;
- set counts;
- exercise and set order;
- category, title, learnings, and timestamps;
- bilateral/unilateral/superset flags;
- modifier and equipment brand;
- export round-trip behavior.

A migration should be repeatable in tests and produce the same persisted IDs and rows for the same fixture database.

### Read-path transition

Switch one read path at a time:

1. workout history;
2. workout detail;
3. exercise progression;
4. statistics;
5. export and Creator OS summary generation;
6. editor save path.

Do not remove legacy storage until all reads and writes use canonical data and at least one compatibility release has validated real upgrades.

## Import and export boundaries

CSV and legacy note input are parsed once into the canonical aggregate. Export reads canonical data.

The Creator OS `TrainingSummary` is a derived projection and is not persisted as another canonical workout table. It can be regenerated from canonical workouts at any time.

## Statistics

Statistics must query canonical typed data. They must not reparse note text during startup or normal operation.

Metric definitions must state handling for:

- unilateral work;
- bodyweight movements;
- duration/distance activities;
- missing weight units;
- missing or ambiguous migrated values;
- sample size and date ranges.

## Rejected alternatives

### Note text remains canonical

Rejected because hidden separators, repeated parsing, parser-version drift, and timestamp identity continue to produce conflicting representations.

### Existing `SetEntity` rows become canonical without workout-exercise rows

Rejected because they lack exercise occurrence identity, explicit ordering, stable workout identity, and reliable source-level compatibility data.

### Big-bang schema/editor rewrite

Rejected because the data-loss risk is too high. Migration, read-path conversion, and editor conversion must be independently reviewable and reversible.

### Use timestamps as IDs

Rejected because a workout can be edited, moved, imported, or share a timestamp with another record. Time is an attribute, not identity.

## Consequences

### Positive

- One source of truth for history, statistics, export, and Creator OS integration.
- Stable identity supports editing, upsert, synchronization, and imports.
- Ordering and exercise occurrences become reproducible.
- Legacy ambiguity is explicit instead of silently normalized.
- Summary sync can remain compact and derived.

### Negative

- Migration is more complex than extending the current set table.
- Compatibility storage temporarily duplicates some data.
- Existing UI and statistics must transition incrementally.
- Category and alias persistence require new tables.
